Stacks

The Most Useful Restricted Data Structure

ES 248 — Data Structures

You Already Use Stacks

Every time you call a function, your program uses a stack.

int factorial(int n) {
    if (n == 0) return 1;
    return n * factorial(n - 1);
}
// factorial(4) calls factorial(3) calls factorial(2) calls factorial(1)
[ factorial(4) | n=4, waiting... ] ← top
[ factorial(3) | n=3, waiting... ]
[ factorial(2) | n=2, waiting... ]
[ factorial(1) | n=1, waiting... ]
[ factorial(0) | returns 1 ] ← currently running
Each call is pushed when started, popped when it returns. The last function called is the first to finish. That is the defining property of a stack.

Stacks Everywhere

Undo / Redo

Every text edit is pushed onto a stack. Ctrl+Z pops the last action. Ctrl+Y re-applies it from a second stack.

VS Code, Word, Photoshop — all stacks.

Browser Back Button

Each page you visit is pushed. Back = pop. Forward = push to a second stack.

The URL bar history is a stack.

Syntax Checking

Your compiler checks that {, (, [ are properly closed. It uses a stack to track which ones are open.

Expression Evaluation

How does a calculator handle 2 + 3 * 4? Operator precedence is managed with a stack — the basis of every compiler.

Pattern: Whenever you need to process something in reverse order of arrival — or remember where you were before going somewhere new — you need a stack.

The Stack — Restricted by Design

A stack is not just an array. It is an array with a contract:

You may only interact with the top.

  • push(x) — add x to the top
  • pop()    — remove and return the top element
  • peek()   — look at the top without removing it
  • isEmpty() — is the stack empty?
LIFO — Last In, First Out. The last thing you pushed is the first thing you get back. The restriction is not a limitation — it is a guarantee that makes problems easier to reason about.

push(1), push(2), push(3) →  top: [ 3 | 2 | 1 ] :bottom

pop() → 3              →  top: [ 2 | 1 ] :bottom

pop() → 2              →  top: [ 1 ] :bottom

Implementing a Stack with an Array

An array + an integer tracking the top index. That is all.

#define MAX 100

struct Stack {
    int data[MAX];
    int top;        // index of the current top element; -1 means empty
};

Visualizing top:

data: [ 10 | 20 | 30 | ?? | ?? | ... ]

          0    1    2    3

                      ↑ top = 2

Initialization: s.top = -1 means the stack is empty — no element at index 0 yet. The condition s.top == -1 is our isEmpty check.

Stack Operations — Code

void init(struct Stack *s) {
    s->top = -1;
}

int isEmpty(struct Stack *s) {
    return s->top == -1;
}

int isFull(struct Stack *s) {
    return s->top == MAX - 1;
}

void push(struct Stack *s, int x) {
    if (isFull(s)) { printf("Stack overflow\n"); return; }
    s->top++;
    s->data[s->top] = x;
}

int pop(struct Stack *s) {
    if (isEmpty(s)) { printf("Stack underflow\n"); return -1; }
    int val = s->data[s->top];
    s->top--;
    return val;
}

int peek(struct Stack *s) {
    if (isEmpty(s)) { printf("Empty stack\n"); return -1; }
    return s->data[s->top];
}

Push and Pop — Step by Step

push(10), push(20), push(30)

After push(10):  [ 10 ]   top=0
After push(20):  [ 10 | 20 ]   top=1
After push(30):  [ 10 | 20 | 30 ]   top=2

pop(), pop()

pop() → 30   [ 10 | 20 ]   top=1
pop() → 20   [ 10 ]   top=0
pop() → 10   []   top=-1 (empty)
Push = increment top, write.   Pop = read, decrement top.   Both are O(1). Always.
Overflow: push when full (top == MAX-1) — writes past the array.   Underflow: pop when empty (top == -1) — reads garbage. Always check.

Problem 1 — Balanced Brackets

Your editor highlights unmatched brackets. How does it work?

{ ( [ ] ) }  →  valid

{ ( [ ) ] }  →  invalid — wrong closing order

{ ( [ ] }   →  invalid — missing )

Key insight: When you see a closing bracket, it must match the most recently opened bracket. "Most recently opened" = top of a stack.

Algorithm:

  • Scan left to right
  • Open bracket ((, [, {) → push it
  • Close bracket (), ], }) → pop and check if it matches
  • End of string → valid only if stack is empty

Balanced Brackets — Walkthrough

Input:  { ( [ ] ) }

Read {   → push    stack: [ { ]
Read (   → push    stack: [ { ( ]
Read [   → push    stack: [ { ( [ ]
Read ]   → pop [    match!    stack: [ { ( ]
Read )   → pop (    match!    stack: [ { ]
Read }   → pop {    match!    stack: []
Stack empty at end → VALID

Balanced Brackets — Code

int isMatching(char open, char close) {
    return (open == '(' && close == ')') ||
           (open == '[' && close == ']') ||
           (open == '{' && close == '}');
}

int isBalanced(char *expr) {
    struct Stack s;
    init(&s);

    for (int i = 0; expr[i] != '\0'; i++) {
        char c = expr[i];
        if (c == '(' || c == '[' || c == '{') {
            push(&s, c);
        } else if (c == ')' || c == ']' || c == '}') {
            if (isEmpty(&s)) return 0;         // nothing to match
            if (!isMatching(pop(&s), c)) return 0;
        }
    }
    return isEmpty(&s);   // valid only if all opens were closed
}
O(n) time — one pass through the string. O(n) space — at most n/2 opens on the stack at once. No nested loops. The stack eliminates the need to search backward.

Problem 2 — Next Greater Element

Given an array, for each element find the next element to its right that is greater.

Input: [ 2, 1, 5, 3, 6, 4 ]

Output: [ 5, 5, 6, 6, -1, -1 ]

For 2: next greater is 5.   For 1: next greater is 5.   For 6: none, so -1.

Brute force: For each element, scan right until you find something larger.

for (int i = 0; i < n; i++) {
    result[i] = -1;
    for (int j = i + 1; j < n; j++) {
        if (arr[j] > arr[i]) { result[i] = arr[j]; break; }
    }
}
O(n²). For n = 100,000 that is 10 billion comparisons. There is a better way.

Next Greater Element — Stack Insight

Scan left to right. Use a stack to track elements waiting for their answer.

Invariant: The stack always holds elements in decreasing order (bottom to top). When a new element arrives that is larger than the top, it is the answer for everything it beats.

Array: [ 2, 1, 5, 3, 6, 4 ]

Read 2:  stack empty, push 2.                Stack: [2]
Read 1:  1 < 2, push 1.                    Stack: [2, 1]
Read 5:  5 > 1, pop 1 → answer[1]=5.     Stack: [2]
         5 > 2, pop 2 → answer[0]=5.     Stack: []
         push 5.                            Stack: [5]
Read 3:  3 < 5, push 3.                    Stack: [5, 3]
Read 6:  6 > 3, pop 3 → answer[3]=6.     Stack: [5]
         6 > 5, pop 5 → answer[2]=6.     Stack: []
         push 6.                            Stack: [6]
Read 4:  4 < 6, push 4.                    Stack: [6, 4]
End:     remaining stack {6, 4} → answer = -1.

Next Greater Element — Code

void nextGreater(int arr[], int result[], int n) {
    struct Stack s;
    init(&s);

    for (int i = 0; i < n; i++) {
        // Pop everything the current element beats — it's their answer
        while (!isEmpty(&s) && arr[peek(&s)] < arr[i]) {
            int idx = pop(&s);
            result[idx] = arr[i];
        }
        push(&s, i);   // push index, not value — so we can write result[idx]
    }

    // Anything still in the stack has no greater element
    while (!isEmpty(&s))
        result[pop(&s)] = -1;
}
SolutionTimeSpace
Brute force (nested loops)O(n²)O(1)
Monotonic stackO(n)O(n)
Each element is pushed exactly once and popped exactly once. Total work = 2n operations = O(n). The stack is not just a container here — it is an active part of the algorithm.

When to Reach for a Stack

The signal: You need to process something in reverse order of arrival — or you need to remember where you were before going somewhere new.

Matching

Balanced brackets, HTML tag matching, XML validation.

Open → push. Close → pop and check.

Monotonic

Next greater/smaller element, stock span, largest rectangle.

Maintain a sorted invariant. Pop when it breaks.

Tracking State

Undo/redo, browser history, function calls, DFS traversal.

Push when entering. Pop when returning.

Monotonic stack is the non-obvious one. Whenever a brute-force solution uses a nested loop to look backward or forward for a smaller/larger element, a monotonic stack can usually reduce it to O(n).

Recap

  • LIFO — the restriction is the point, not a limitation
  • Array implementation — a struct with data[] and top. Push/pop are O(1).
  • Balanced brackets — O(n) with a stack, because "most recent open" = top
  • Monotonic stack — turns O(n²) nested-loop problems into O(n)
